# AUDIT — 2026-03-07 (v3)

## Project Context

**wain** is a statically-compiled Go UI toolkit with GPU rendering via Rust. The project targets developers building hardware-accelerated UI applications who need single-binary deployment (no runtime dependencies), direct GPU access without heavyweight frameworks, and native X11/Wayland protocol support.

**Current Phase:** Phase 1 (Software Rendering Path) at **85% completion**. The project has successfully implemented Wayland/X11 protocol clients, a software 2D rasterizer, and a UI widget layer. GPU rendering (Phases 2-8) remains unimplemented.

**Target Audience:** Linux developers requiring fully static binaries with direct protocol access for embedded systems, containers, or minimal deployment environments.

### MEDIUM

- [ ] **High-complexity layout functions lack decomposition** — internal/ui/layout/layout.go:209,319 — Functions `layoutRow` (92 lines, CC=17) and `layoutColumn` (92 lines, CC=17) exceed recommended complexity thresholds (target: CC ≤10, length ≤50 lines). Both implement multi-phase layout algorithms (measure children, compute flex distribution, calculate positions, apply alignment) in monolithic blocks. The identical structure (both 92 lines, CC=17) indicates ~95% code duplication. Extracting shared logic into helper functions (`measureChildren()`, `distributeFlexSpace()`, `applyAlignment()`) would reduce complexity to target levels. Risk: Future enhancements (grid layout, min/max constraints, aspect ratio) will compound existing complexity. Both functions handle 8+ control flow branches for flex-grow, flex-shrink, gap spacing, padding, and alignment modes.

- [ ] **Wire protocol encoding has high cyclomatic complexity** — internal/wayland/wire/wire.go:333 — Function `EncodeMessage` has cyclomatic complexity 17 across 68 lines, handling marshaling for 9 Wayland wire format types (int32, uint32, fixed, string, object, new_id, array, fd, pad) with nested error handling. This is protocol-critical code where bugs manifest as compositor crashes or protocol violations. The function correctly implements the Wayland wire protocol specification, but its complexity makes it difficult to review for edge cases. No fuzz tests were detected in `wire_test.go` to verify robustness against malformed input. Recommendation: Add fuzz tests using `go test -fuzz` to validate wire format encoding/decoding handles corrupted or malicious input safely. Similar complexity exists in X11 wire encoding (DecodeSetupReply: 98 lines, CC=11).

- [ ] **Phase 2+ GPU roadmap extensive but unimplemented** — ROADMAP.md:79-306, README.md:49,309-312 — The project documentation describes an 8-phase roadmap with detailed plans for DRM/KMS buffer infrastructure (Phase 2, 48 lines), Intel GPU command submission (Phase 3, 52 lines), shader compiler pipeline (Phase 4, 41 lines), GPU rendering backend (Phase 5, 28 lines), AMD GPU support (Phase 6, 43 lines), and polish features (Phases 7-8, 44 lines). Code search across the repository finds zero occurrences of DRM, KMS, GPU command buffers, Vulkan, or Mesa in implementation files. The Rust render-sys library (44 lines total in src/lib.rs) implements only two C ABI test functions (`render_add`, `render_version`). No evidence exists of DRM ioctl wrappers, batch buffer construction, shader compilation, or GPU-specific code. While the roadmap correctly marks these as "Planned" and the README states they are "Not yet implemented," the extensive roadmap detail (228 lines covering 6 future phases) may create unrealistic expectations about implementation proximity. The roadmap is well-researched and architecturally sound, but represents future work that is not yet started.

- [ ] **No integration tests for protocol stack** — internal/wayland/, internal/x11/, internal/raster/ — The repository has comprehensive unit tests (19 packages with tests, all passing) but no end-to-end integration tests covering the protocol → rasterizer → display pipeline. Unit tests verify individual components (wire encoding, memory allocation, rendering primitives), but integration bugs could exist at component boundaries. For example, no test verifies that a complete Wayland connection sequence (connect → registry → compositor → surface → shm pool → buffer → commit) works end-to-end. The `cmd/demo` binary (181 lines) provides manual verification but is not automated. Recommendation: Add integration tests that either use a headless compositor (weston-headless) or mock the compositor side to verify the full client stack. This would catch issues like incorrect event sequencing, buffer lifecycle bugs, or fd leakage that unit tests miss.

### LOW

- [ ] **gen-atlas tool undocumented in README** — cmd/gen-atlas/main.go:1-142, README.md — The repository contains a second binary (`cmd/gen-atlas`, 142 lines) that generates SDF (Signed Distance Field) font atlases for the text rasterizer (`internal/raster/text/`). This tool is referenced in ROADMAP Phase 1.4 ("SDF-based text rendering using a pre-baked SDF font atlas embedded in the binary") and is critical for regenerating embedded font data if users want to change fonts or add glyphs. However, the README has no mention of this tool in the Build, Run, or Development sections. Users discovering the SDF text rendering capability will not know how the atlas is generated or how to customize it. The tool has no tests (`[no test files]`) and no Makefile target. Recommendation: Add documentation explaining what gen-atlas does, when to use it, and how to run it. Add a basic smoke test to verify the tool builds and produces valid output.

- [ ] **No build verification for gen-atlas** — Makefile:1-135 — The Makefile defines targets for `build`, `test`, `test-go`, `test-rust`, `check-static`, and `demo` but no target for building or testing `cmd/gen-atlas`. The `go list ./...` output confirms the package exists but `go test ./...` reports `[no test files]`. This means the atlas generator could break without CI detection. The tool is critical infrastructure for the text rendering subsystem — if it fails, the embedded font atlas cannot be regenerated, making font customization impossible. Recommendation: Add `make gen-atlas` target to build the tool and at minimum a smoke test to verify it produces valid atlas output (non-zero file size, valid header magic). Consider adding the build to CI to catch regressions. Estimated effort: 30 minutes for Makefile target + basic test.

## Risk Assessment

### Immediate Risks (Impact: Low, Probability: Low)

**None identified.** All critical functionality is implemented and tested. No blocking issues exist for Phase 1 completion.

### Technical Debt (Impact: Medium, Probability: Medium)

1. **Layout algorithm complexity** — The `layoutRow` and `layoutColumn` functions (92 lines, CC=17 each) will be difficult to extend. Future enhancements (grid layout, min/max constraints, aspect ratio) will compound existing complexity. Refactoring is recommended before adding features. Impact is medium because layout is a core system that will require frequent modification.

2. **No wire protocol fuzzing** — The Wayland and X11 wire format encoders/decoders lack fuzz tests. While unit tests exist, malformed input handling is untested. Risk is moderate because protocol bugs cause compositor crashes or security vulnerabilities. Recommendation: Add `go test -fuzz` tests for EncodeMessage/DecodeMessage to find edge cases.

3. **gen-atlas tool is untested** — The SDF atlas generator has no tests or documentation. If it breaks, users cannot regenerate embedded font data. Risk is low because the tool is rarely used (atlas is pre-baked), but CI should catch regressions. Impact increases if users need custom fonts.

4. **No integration tests** — While unit tests are comprehensive (19 packages, all passing), no end-to-end tests verify the protocol → rasterizer → display pipeline. Integration bugs at component boundaries would not be detected until manual testing. Recommendation: Add integration tests using headless compositor or mocks.

### Long-term Concerns (Impact: Low, Probability: Low)

1. **GPU roadmap ambiguity** — The 8-phase roadmap devotes 228 lines to GPU rendering details (DRM, Intel GPU, shaders, AMD) but none of this code exists. If GPU work is deprioritized, the roadmap should be updated to manage expectations. Current status is accurate (clearly marked "Planned," README states "Not yet implemented") but may mislead about implementation proximity. The roadmap is architecturally sound and well-researched, making it valuable documentation even without implementation.

2. **Internal-only packages** — All 18 packages are in `internal/`, preventing external use. This is intentional (per README: "no public API surface yet") but limits the project's utility until Phase 1 completion. No risk, just a documented limitation.
