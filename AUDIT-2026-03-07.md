# AUDIT — 2026-03-07

## Project Context

**wain** is described as "A statically-compiled Go UI toolkit with GPU rendering via Rust" targeting developers who need hardware-accelerated UI applications with single-binary deployment, direct GPU access, and native X11/Wayland protocol support.

**Current Phase:** README.md claims the project is in **Phase 0** (Foundation & Toolchain Setup) with only basic Go → Rust static library linking implemented. The README explicitly states: **"Not yet implemented: GPU rendering, Mesa/Vulkan integration, X11/Wayland protocol support, or UI toolkit APIs are planned for future phases."**

**Repository Structure:** 22 Go packages, 61 Go source files, 17,941 total lines of code, 350 functions analyzed.

## Summary

**Overall Health:** The codebase is technically sound (all tests pass, go vet clean, proper static linking verified), but suffers from **CRITICAL documentation discrepancy**. The README is severely outdated and misrepresents the actual implementation status.

## Findings

### CRITICAL

- [x] **README claims Phase 0 but implements through Phase 1.5** — README.md:17-18, 147-187 — The README states "Not yet implemented: GPU rendering, Mesa/Vulkan integration, X11/Wayland protocol support, or UI toolkit APIs are planned for future phases" but the codebase contains 342/350 functions (97.7%) implementing features from Phases 1.1-1.5 of the ROADMAP. Specifically: (a) Full Wayland client protocol implementation (wl_display, wl_registry, wl_compositor, wl_surface, wl_shm, xdg_wm_base, xdg_surface, xdg_toplevel) in internal/wayland/; (b) Full X11 protocol implementation (CreateWindow, MapWindow, CreateGC, PutImage) in internal/x11/; (c) Input handling (wl_seat, wl_pointer, wl_keyboard) in internal/wayland/input/; (d) Complete software rasterizer (rectangles, rounded rects, curves, gradients, box shadows, SDF text rendering, alpha compositing) in internal/raster/; (e) Widget layer (Button, TextInput, ScrollContainer) with layout system in internal/ui/. This creates a massive gap between documented behavior and actual capabilities, preventing users from discovering or utilizing ~98% of the implemented functionality. **FIXED:** README.md updated to document Phase 1 completion (85%), list all implemented protocol/rendering/UI features, add detailed architecture section, document known limitations, and update contributing guidelines.

### HIGH

_No HIGH severity findings._

### MEDIUM

- [ ] **No demonstration binary for Phase 1 features** — cmd/wain/main.go:1-21 — The main binary (cmd/wain) only demonstrates Phase 0 functionality (render.Add, render.Version) despite full implementation of Wayland/X11 clients, rasterizer, and widgets. ROADMAP.md Phase 1.5 claims "Milestone: interactive demo app (text fields, buttons, scrolling list) running on software renderer over both X11 and Wayland" but no such binary exists. Users cannot verify the 97.7% of implemented features without writing integration code themselves. Evidence: cmd/wain/main.go contains only 20 lines, calls only render.Add and render.Version; no reference to internal/wayland, internal/x11, internal/raster, or internal/ui packages.

- [ ] **Undocumented exported interface methods** — internal/wayland/client/client.go:80,84; internal/wayland/input/input.go:47,51; internal/wayland/shm/shm.go:32,36; internal/wayland/xdg/xdg.go:41,45; internal/wayland/wire/wire.go:422 — 9 exported functions/methods lack godoc comments: client.Object.ID(), client.Object.Interface(), input.Seat.ID(), input.Seat.Interface(), shm.Pool.ID(), shm.Pool.Interface(), xdg.Surface.ID(), xdg.Surface.Interface(), wire.Writer.Write(). These are interface implementations that should document their contract. Doc coverage drops from 100% to 96.9% due to these omissions.

- [ ] **High-complexity layout functions lack decomposition** — internal/ui/layout/layout.go:209,319 — Functions layoutRow and layoutColumn both have cyclomatic complexity 17 and 107 lines of code. These exceed recommended thresholds (complexity >15, length >50) and handle multiple layout phases (measurement, distribution, alignment) in monolithic implementations. Both functions have 11 nested control structures and share identical complexity patterns, suggesting opportunity for shared helper extraction. Risk: future layout algorithm changes (e.g., adding min/max constraints, aspect ratio) will increase complexity further.

- [ ] **Phase 2+ GPU claims remain unimplemented** — ROADMAP.md:79-306 — ROADMAP describes detailed plans for DRM/KMS buffer infrastructure (Phase 2), Intel GPU command submission (Phase 3), shader compiler pipeline (Phase 4), and AMD GPU support (Phase 6). Grep search for "DRM", "KMS", "GPU", "Vulkan", "Mesa" across codebase returns zero matches in non-test Go files. The render-sys Rust library (185 total lines) only implements trivial C ABI test functions (render_add, render_version). No evidence of DRM ioctl wrappers, batch buffer construction, or shader compilation infrastructure. The 8-phase roadmap sets expectations that ~20% of the work is GPU-related, but 0% is implemented.

- [ ] **Wire protocol encoding function has high cyclomatic complexity** — internal/wayland/wire/wire.go:333 — Function EncodeMessage has cyclomatic complexity 17 across 81 lines, handling marshaling for 9 Wayland wire format types (int32, uint32, fixed, string, object, new_id, array, fd, pad) with nested error handling. This is a protocol-critical function where bugs cause compositor protocol errors. No fuzzing detected for wire format (no fuzz tests in wire_test.go). Recommendation: Add fuzz tests for EncodeMessage/DecodeMessage to detect malformed input handling.

### LOW

- [ ] **No build verification for gen-atlas** — Makefile:1-135 — The Makefile defines targets for `build`, `test`, `test-go`, `test-rust`, and `check-static` but no target for building or testing cmd/gen-atlas. The `go list ./...` output shows github.com/opd-ai/wain/cmd/gen-atlas has [no test files]. Recommendation: Add `make gen-atlas` target and at least smoke test to verify the tool builds.

## Risk Assessment

### Immediate Risks (Impact: High, Probability: High)

1. **User confusion due to documentation mismatch**: New users will read README, conclude project only has Phase 0, and not discover the extensive Wayland/X11/rasterizer/widget implementations. This undermines adoption and contribution.

2. **Wasted implementation effort**: 342 functions implementing sophisticated graphics protocols, rasterization algorithms, and UI widgets are effectively hidden. Contributors may duplicate work thinking features aren't implemented.

### Technical Debt (Impact: Medium, Probability: Medium)

1. **No integration testing for Phase 1 stack**: Individual packages have unit tests (all passing), but no end-to-end test demonstrates Wayland connection → surface creation → software rendering → display. The absence of a working demo means integration bugs are undetected.

2. **Layout algorithm complexity**: The layout system (layoutRow, layoutColumn) will be difficult to extend. Adding features like grid layout, aspect ratio constraints, or percentage-based sizing will compound complexity.

### Long-term Concerns (Impact: Low, Probability: Low)

1. **Roadmap-code drift**: If Phase 2+ GPU work begins, the gap between documentation and reality may widen further. The project needs a documentation update process.

## Recommendations

### Priority 1 (Address CRITICAL finding)

1. **Update README.md to reflect actual implementation status**: Document Phase 1.1-1.5 completion in the "Current Functionality" section. Add usage examples for Wayland/X11 clients, rasterizer, and widgets. Update the "Not yet implemented" disclaimer to only list Phases 2+.

2. **Create Phase 1 demonstration binary**: Build cmd/wain-demo that opens a window (X11 or Wayland), renders UI widgets (button, text input, scroll container), handles input events, and demonstrates the complete Phase 1 stack. This serves as integration test and proof of implementation.

### Priority 2 (Address MEDIUM findings)

1. **Document undocumented exported symbols**: Add godoc comments to the 9 interface methods (ID, Interface, Write). Even if they're interface implementations, document the contract.

2. **Refactor layout complexity**: Extract shared logic from layoutRow and layoutColumn into helper functions (e.g., measureChildren, distributeSpace, alignItems). Target cyclomatic complexity <10 per function.

3. **Add fuzzing for wire protocols**: Implement fuzz tests for internal/wayland/wire and internal/x11/wire to catch malformed input handling bugs before compositor integration.

4. **Document gen-atlas tool**: Add section to README explaining SDF atlas generation workflow, when to regenerate, and customization options.

### Priority 3 (Address LOW findings, improve quality)

1. **Add gen-atlas to Makefile**: Create `make gen-atlas` target, ensure it builds during CI.

2. **Improve layout test coverage**: Add tests for edge cases (zero-width children, negative spacing, conflicting constraints) to layout_test.go.

3. **Consider roadmap revision**: If GPU work (Phases 2-6) is deprioritized, update ROADMAP.md to reflect focus on software rendering path as primary feature.

## Conclusion

The wain codebase is **technically well-implemented** with clean architecture, good test coverage, proper static linking, and low average complexity. However, it suffers from a **critical documentation discrepancy** that renders 97.7% of its implemented features undiscoverable to users and contributors.

**The project is not in Phase 0.** It has completed Phase 1.1-1.5 (Wayland/X11 protocols, software rasterizer, widget layer) but the README claims otherwise. This is the audit's primary finding and must be corrected immediately.

**Action Required:** Update README.md and create a working demonstration binary before considering the project "Phase 0 complete." The code quality is high, but the documentation must match reality.
